\section*{Introduction}
In this lab we set up to compare different implementations of histogram 
generation on GPU.  The histogram is a fundamental data structure that counts the frequency
of appearances of a certain data value in a dataset, commonly used to estimate 
the probability distribution of such data. Histograms find applications 
in computer vision, data analytics and scientific computing. The fundamental
challenge of histogram generation on GPU is that when multiple threads
attempt to update the same memory location (bin) a memory conflict may occur.
Avoiding this conflicts are the main optimization ideas we will explore in this lab,
followed by efficient data communication and probably efficient encoding.

In this Lab we have coded multiple implementations of histogram generation,
from a global memory version with atomic operations (V0),
to a shared memory block-privatized version (V1),
to a vectorized $\mathcal{R}$=2-per-block version with RLE compression (V2, the GOAT),
to an ablation of V2 without RLE (V3),
to a warp-aggregated version (V4),
to a bin-centric gather version with zero atomics (V5),
to a sort-and-reduce approach via Thrust (V6),
and finally to a multi-split hierarchical bucketing version (V7).

\section{Analysis 1: V0 vs.\ V1 — Global vs.\ Shared Memory}
% ════════════════════════════════════════════════════════════

\subsection*{The ``Too Small'' Dataset and the P100 L2 Cache}

A critical architectural observation shapes the results in this lab:
$N=500{,}000$ integers occupy only 2\,MB of device memory, which fits
\emph{entirely} within the P100's 4\,MB L2 cache.
On legacy architectures, 500{,}000 threads hammering a single global address
(Experiment 2) would cause catastrophic serialisation lasting tens of
milliseconds.
On the P100, the L2 cache's hardware atomic resolution unit resolves
collisions on-chip, masking much of the global memory bottleneck.
This architectural shield makes the absolute penalty for V0 under uniform
data far smaller than textbook analysis predicts, and is the primary reason
all versions cluster in a narrow sub-millisecond band.

\subsection*{V0 vs.\ V1 on Experiment 1 (random data)}

With random data, 500{,}000 elements distribute uniformly across 4096 bins
($\approx$122 hits per bin), yielding low per-bin contention.
V0 (\EOneVZeroAvg~ms) and V1 (\EOneVOneAvg~ms) are nearly tied.
The speedup of V1 over V0 is $\EOneVZeroAvg/\EOneVOneAvg \approx 1.03\times$,
confirming that shared-memory privatization offers little benefit when global
atomic queues are already short.
Reference Figure~\ref{fig:exp1_speedup}.

\subsection*{V0 vs.\ V1 on Experiment 2 (uniform data)}

All 500{,}000 elements target bin 42, concentrating all atomic traffic on a
single address.
V0 degrades to \ETwoVZeroAvg~ms --- a $1.68\times$ slowdown over its
random-data baseline (\EOneVZeroAvg~ms).
V1 (\ETwoVOneAvg~ms) remains nearly unchanged because block privatization
resolves collisions on-chip in shared memory; only one global atomic fires
per block per non-zero bin at the merge step.
The V1 advantage over V0 is $\ETwoVZeroAvg/\ETwoVOneAvg \approx 1.71\times$
under uniform contention, vs.\ a negligible $1.03\times$ on random data ---
the advantage is clearly \emph{larger} in Experiment 2.
Reference Figure~\ref{fig:exp2_speedup}.

\subsection*{Impact of input distribution on V0}

V0 degrades from \EOneVZeroAvg~ms (random) to \ETwoVZeroAvg~ms (uniform),
a $1.68\times$ slowdown caused by all $N$ global atomics targeting one bin.
Without the P100 L2 cache absorbing the traffic on-chip, the same test on
earlier architectures would show a 10--50$\times$ penalty.
Reference Figure~\ref{fig:comparison}.

\subsection*{Impact of input distribution on V1}

V1 remains essentially flat: \EOneVOneAvg~ms (random) vs.\
\ETwoVOneAvg~ms (uniform), a difference of less than 1\%.
Shared-memory atomics resolve collision overhead at the SM level, and
uniform data ($K=1$ non-zero bin) actually \emph{reduces} the global merge
cost compared to random data's $K\approx4096$ non-zero bins.
This flat response, contrasting sharply with V0's $1.68\times$ degradation,
confirms that block privatization decouples performance from data distribution.

\section{Analysis 2: Version 2 — Vectorized $\mathcal{R}$-per-Block Privatization}
% ════════════════════════════════════════════════════════════

\subsection*{Implementation approach}

Version 2 combines \emph{$\mathcal{R}$-per-Block Histogram Replication} ($\mathcal{R}=2$)
with \emph{Vectorized 128-bit Loads}.
Two copies of the shared-memory histogram (\texttt{multi\_histo[0]} and
\texttt{multi\_histo[1]}) are allocated, each padded by one element (4097
\texttt{unsigned int}s per copy) to break the 32-bank stride alignment that
arises when \texttt{num\_bins} is a power of two.
Even-indexed threads scatter into copy~0; odd-indexed threads scatter into
copy~1, so every 32-thread warp splits its atomic traffic evenly across two
physical bank regions.
After the accumulation phase, a single \texttt{\_\_syncthreads()} separates
the per-block merge: each thread sums its two copies and issues \emph{one}
global \texttt{atomicAdd} only when the total is non-zero (sparsity filter).
Simultaneously, V2 casts the 32-bit input array to \texttt{uint4}, loading
four integers per 128-bit global memory transaction and quadrupling effective
bandwidth over V0/V1's scalar 32-bit reads.

\subsection*{Superiority over V0 and V1}

V0 issues $N$ \emph{global} atomic operations at DRAM latency.
V1 migrates these to low-latency shared memory but uses a single histogram
copy: in a 32-wide warp, if $k$ threads map to the same 32-wide bank group,
those $k$ atomics are serialised in hardware.
V2 halves that contention by distributing 32 warp lanes across two buffer
copies, reducing the maximum same-bank collision count from 32 (V1) to 16.
The $+1$ padding ensures the two arrays start on different bank offsets,
preventing the second copy from merely replicating the same conflict pattern.
For the global flush, the sparsity filter eliminates all-zero bins entirely,
reducing PCIe/L2 pressure at the per-block merge step.

\subsection*{Performance vs.\ data distribution}

\begin{itemize}[noitemsep]
  \item \textbf{Random (Experiment 1):} Bank conflicts are already rare when
        4096 bins are addressed randomly; both the \texttt{uint4} bandwidth gain
        and the sparsity filter contribute.
        See Figure~\ref{fig:exp1_speedup}.
  \item \textbf{Uniform (Experiment 2):} All 32 warp lanes target the same bin.
        In V1 all 32 hit the same bank — 32-way serialisation.
        In V2 they split: 16 go to \texttt{multi\_histo[0]} and 16 to
        \texttt{multi\_histo[1]}, immediately halving the hardware serialisation
        penalty.  The sparsity filter then ensures exactly \emph{one} global
        atomic fires at the end, regardless of how many thousands of identical
        elements the block processed.
        See Figure~\ref{fig:exp2_speedup}.
  \item \textbf{Static shared memory footprint:}
        $2 \times 4097 \times 4\,\text{B} = 32{,}776\,\text{B}$, well within
        P100's 64\,KB shared memory limit per SM.
\end{itemize}

The measured data confirms these predictions:
\begin{itemize}[noitemsep]
  \item \textbf{Random (Exp 1):} V2 (\EOneVTwoAvg~ms) vs.\ V1
        (\EOneVOneAvg~ms) --- a gain of only $\approx1\%$.
        The \texttt{uint4} loads and $\mathcal{R}{=}2$ bank-conflict mitigation
        provide a small consistent improvement; RLE provides almost none
        because consecutive identical bins are rare (probability $1/4096$).
  \item \textbf{Uniform (Exp 2):} V2 (\ETwoVTwoAvg~ms) achieves a
        $1.73\times$ speedup over V0 (\ETwoVZeroAvg~ms).
        Surprisingly, V2 and V1 (\ETwoVOneAvg~ms) are nearly tied even under
        maximum RLE compressibility --- see Analysis~3 for the ablation
        interpretation.
\end{itemize}

% ════════════════════════════════════════════════════════════
\section{Analysis 3: Version 3 — Ablation: R=2 Without RLE}
% ════════════════════════════════════════════════════════════

\subsection*{Implementation approach}

Version 3 is a controlled ablation of Version 2.
It retains the full $\mathcal{R}=2$ per-block replication structure,
the $+1$-padded static shared memory (\texttt{multi\_histo[2][4097]}),
the \texttt{uint4} 128-bit vectorized load path, and the sparsity-filtered
global atomic flush --- but removes the register-level RLE state machine.
Instead of accumulating consecutive identical bins in a two-register
\texttt{(current\_bin, run\_count)} pair, V3 issues an \texttt{atomicAdd}
to shared memory immediately for every valid element.

\subsection*{Purpose: isolating the RLE contribution}

By holding all other algorithmic choices constant (two copies, padding,
\texttt{uint4}, sparsity filter) and removing only RLE, any measured
performance difference between V2 and V3 is attributable \emph{solely}
to RLE temporal compression.

\subsection*{Expected behaviour}

\begin{itemize}[noitemsep]
  \item \textbf{Random (Experiment 1):} Elements are i.i.d.\@ uniform over
        $[0,4095]$, so consecutive identical values are rare
        (probability $\tfrac{1}{4096}$).  RLE run lengths average just above
        one; V2's state machine fires almost every element anyway.
        V3 and V2 should post nearly identical performance on random data.
  \item \textbf{Uniform (Experiment 2):} All 500{,}000 elements are identical.
        V2's RLE accumulates the entire block's work into a single register
        counter, issuing \emph{one} shared \texttt{atomicAdd} per thread lane
        per block rather than one per element --- dramatically reducing shared
        memory atomic traffic.
        V3 fires 500{,}000 atomics per thread into shared memory, incurring
        maximum serialisation.
        The V2 vs.\ V3 gap should be large for uniform data.
\end{itemize}

\begin{figure}[htbp]
  \centering
  \includegraphics[width=0.65\textwidth]{fig_v2_v3_ablation.pdf}
  \caption{RLE ablation: V3 (R=2, no RLE) vs.\ V2 (R=2+RLE) for both
           experiments.  Difference isolates the pure temporal compression gain.}
  \label{fig:v2v3ablation}
\end{figure}

The measured delta is negligible: V2 vs.\ V3 in Experiment~1 are
\EOneVTwoAvg~ms vs.\ \EOneVThreeAvg~ms ($\Delta < 0.1\%$); in
Experiment~2 they are \ETwoVTwoAvg~ms vs.\ \ETwoVThreeAvg~ms
($\Delta < 0.5\%$, within one standard deviation).
\textbf{RLE compression provides no measurable speedup at $N=500{,}000$ on
the P100}, in either distribution.
The most likely explanation is that the Pascal SM's shared-memory atomic
pipeline is fast enough that the RLE register overhead (per-element branch
and conditional counter) costs as many cycles as it saves in atomic
operations.
The dominant mechanism in V2 is therefore the $\mathcal{R}{=}2$
bank-conflict mitigation and \texttt{uint4} vectorized loads --- not the
temporal register compression.
Reference Figure~\ref{fig:v2v3ablation}.

\section{Additional Kernel Variants}
% ════════════════════════════════════════════════════════════

\subsection*{V4 — Warp Aggregation}

Version 4 introduces intra-warp aggregation using CUDA warp-level
primitives.  For each input element, the thread computes its target bin and
uses \texttt{\_\_ballot\_sync} to identify which other lanes in the same warp
map to the same bin.  One leader (lowest set bit via \texttt{\_\_ffs}) reads
the group count from \texttt{\_\_popc} and issues \emph{one}
\texttt{atomicAdd} with the aggregated value, while the remaining lanes in
the group skip their atomic entirely.
This reduces the number of shared-memory atomic operations by a factor equal
to the average intra-warp bin collision rate.
V4 uses \texttt{extern \_\_shared\_\_ unsigned int histo\_private[]} sized to
\texttt{num\_bins} 32-bit integers at launch time.
The sparsity-filtered global flush is retained from V2.

\textbf{Expected:} On uniform data (Experiment 2), all 32 lanes in a warp
target the same bin every iteration, so the aggregation factor is 32 ---
a full warp reduces to one atomic.  On random data, the average collision
rate is $32/4096 \approx 0.0078$, so the benefit is modest.

\subsection*{V5 — Bin-Centric Gather}

Version 5 inverts the scatter pattern used by V0--V4.
Instead of each thread owning an \emph{input element} and scattering it
to a bin, each thread owns a \emph{bin} and gathers through the input to
count matches.
With \texttt{num\_bins}=4096 and \texttt{blockDim}=256, exactly 16 blocks
are launched and every bin is covered by exactly one thread.
Because no two threads write to the same output location,
\textbf{zero atomic operations} are required anywhere.
Shared-memory tiling amortises global memory latency: each block loads a
256-element tile in one coalesced 1024-byte transaction, and all 256 threads
scan the tile simultaneously in registers.

\textbf{Trade-off:} The elimination of atomics is achieved at the cost of
$O(N \times B)$ comparisons where $B = 4096$.  For $N = 500{,}000$ this
means $2 \times 10^9$ comparison iterations.  Performance is
arithmetic-bound, not memory-bandwidth-bound, and is dominated by loop
count regardless of data distribution.

\subsection*{V6 — Sort \& Reduce-by-Key via Thrust}

Version 6 reframes histogram construction as a sort problem.
The device input array is radix-sorted in-place via
\texttt{thrust::sort}, converting the unordered element stream into a
contiguous sequence of equal values.  A subsequent
\texttt{thrust::reduce\_by\_key} collapses each contiguous run to a
(bin, count) pair in a single pass.
\texttt{thrust::scatter} then places the counts at their correct positions
in the output bin array, and a final \texttt{convert\_kernel} applies the
127-cap clipping.

This approach requires no atomic operations and avoids shared-memory
contention entirely.  However, it incurs an $O(N \log N)$ GPU radix sort
(Thrust uses CUB internally) before the $O(N)$ reduction.
Because \texttt{thrust::sort} modifies \texttt{d\_input} in-place, each
experimental run reloads input from disk to preserve correctness.

\textbf{Expected:} Sort latency dominates.  Likely slower than V1/V2
for random data, but may be competitive with V5 (Gather) for highly skewed
data where the sorted stream is nearly run-length-compressed already.

\subsection*{V7 — Multi-split Hierarchical Bucketing}

Version 7 applies a two-level hierarchical decomposition of the 4096-bin
histogram.  Each 12-bit bin index is split as
$\text{super\_bucket} = b \mathbin{/} 256$ and $\text{offset} = b \bmod 256$,
with the shared-memory target address computed as
$\text{target} = \text{super\_bucket} \times 256 + \text{offset}$
(which equals $b$, preserving correctness while encoding a locality hint in
the access pattern).

With 16 super-buckets of 256 bins each, threads in a warp that hash to the
same super-bucket compete for nearby shared-memory addresses within one
256-element region, improving cache-line reuse and reducing the probability
of cross-region bank conflicts compared to a flat 4096-bin address space.
V7 uses \texttt{extern \_\_shared\_\_ unsigned int histo\_private[]} (same
dynamic size as V4) and the sparsity-filtered global flush.

\section{Short Answer Questions}
% ════════════════════════════════════════════════════════════

\subsection*{Global memory reads per kernel}

All scatter-based kernels (V0--V4, V7) read the input array exactly once ---
each of the $N$ elements is loaded by exactly one thread.
V5 (Bin-Centric Gather) effectively reads the entire input for each of $B$
bins, yielding $O(N \times B)$ global load operations.
V6 reads the input once for \texttt{thrust::sort}, which then operates
in-place.
\textbf{Global reads: $N$ for V0, V1, V2, V3, V4, V6, V7; $O(N \times B)$ for V5.}

\subsection*{Global memory writes per kernel}

\begin{itemize}[noitemsep]
  \item \textbf{V0:} Every \texttt{atomicAdd(\&bins[b],\,1)} is a global
        read-modify-write.  \textbf{$N$ global atomic writes total.}
  \item \textbf{V1, V2, V3, V4, V7:} Accumulation uses shared memory.
        The merge phase issues at most $\text{BLOCKS} \times K$ global
        atomic writes, where $K \leq 4096$ is the number of non-zero bins.
        For random data $K \approx 4096$; for uniform data $K = 1$.
  \item \textbf{V5:} One non-atomic write per bin: exactly $B = 4096$ global
        writes total, regardless of $N$ or distribution.
  \item \textbf{V6:} \texttt{thrust::sort} operates in-place on $d\_input$;
        \texttt{thrust::scatter} writes at most $N_{\text{unique}} \leq B$
        locations to the bin array.
\end{itemize}

\subsection*{Atomic operations per kernel}

\begin{itemize}[noitemsep]
  \item \textbf{V0:} $N$ global atomics --- one per element.
  \item \textbf{V1:} $N$ shared atomics (accumulation) $+$ up to
        $\text{BLOCKS} \times K$ global atomics (merge).
        Total: $N + \text{BLOCKS} \cdot K$.
  \item \textbf{V2:} $N$ shared atomics split across $\mathcal{R}{=}2$ copies
        (RLE reduces the effective count for long runs) $+$ up to
        $\text{BLOCKS} \times K$ global atomics (sparsity-filtered merge).
  \item \textbf{V3:} Same structure as V2 without the RLE reduction step ---
        all $N$ shared atomics fire unconditionally.
  \item \textbf{V4:} $N/\text{agg}$ shared atomics (warp-aggregated, where
        $\text{agg}$ is the average same-bin collision count per warp) $+$
        up to $\text{BLOCKS} \times K$ global atomics (sparsity-filtered merge).
  \item \textbf{V5:} \textbf{Zero} --- no \texttt{atomicAdd} anywhere.
        Each bin is owned by exactly one thread; no concurrent writes occur.
  \item \textbf{V6:} \textbf{Zero} post-sort --- Thrust radix sort and
        reduce-by-key use their own internal synchronisation primitives,
        not user-level atomic operations.
  \item \textbf{V7:} $N$ shared atomics (same count as V1; locality hint
        does not reduce the number of atomics, only their distribution
        within the shared-memory address space) $+$ up to
        $\text{BLOCKS} \times K$ global atomics (sparsity-filtered merge).
\end{itemize}
